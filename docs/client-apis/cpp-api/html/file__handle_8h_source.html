25// FileHandle is the in-memory representation of the file. It holds an open file descriptor to the
26// file as well as the state of each page in the file - if it is pinned in the Buffer Manager. File
27// Handle is the bridge between a Column/Lists/Index and the Buffer Manager that abstracts the file
28// in which that Column/Lists/Index is stored.
29// Warning: FileHandle is *NOT* thread-safe. That is multiple parallel threads can access its
30// functions that are used by query processor, such as getFrameIdx, swizzle, unswizzle,
31// acquirePageLock, and releasePageLock. Similarly, functions that change the internal data
32// structures, such as addPage and resetToZeroPagesAndPageCapacity can be accessed concurrently.
33// However, the query processor functions and those that change the internal structures *CANNOT* be
34// called concurrently. The caller needs to synchronize these calls.

65 // Warning: Adding a new page does not write a new page directly to a file. Instead, it only
66 // creates a pageLock and a spot in the pageIdxToFrameMap. This may have dangerous consequences
67 // if a concurrent thread attempts to read the page. This should be ensured by the caller. To be
68 // more specific, let the T_W be the writer thread that is adding the page and T_R a concurrent
69 // reader thread. The sequence of calls to make sure the page can be read and written safely is
70 // this: (1) T_W calls addNewPage() to create a new page, e.g., with pageIdx: 7 (2) T_W calls
71 // BufferManager::pinWithoutReadingFromFile(thisFileHandle, 7 /* pageIdx */, ...):
72 // pinWithoutReadingFromFile will not try to read the page from the file, so this is safe.
73 //
74 // After (1) and (2) T_R can now pin pageIdx 7 by calling BufferManager::pin(...).
75 //
76 // If this sequence is not enforced, there can be serious side effects, including T_R trying to
77 // pin a new page, which will trigger the BM to read offsets in the file on disk, where those
78 // offsets do not yet exist, which can cause the OS to error. In general code that uses
79 // addNewPage should be written to ensure that once T_W calls addNewPage, it should be followed
80 // by a BufferManager::pinNewPage call and the code should ensure that no other thread can try
81 // to pin the newly added page in between these calls.
